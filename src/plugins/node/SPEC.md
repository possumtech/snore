# SPEC.md — Node Plugin Specification

## 0. Overview

The `node` plugin provides integrated Node.js development functionality
(testing, profiling, tracing, coverage) through a single multiplexed
namespace: `node://`.

It follows the Rummy principle of **proposing actions** (server-side) and
**executing side-effects** (client-side).

## 1. Unified Namespace: `node://`

All artifacts generated by this plugin use the `node://` scheme.

| Sub-path | Fidelity | Purpose |
| :--- | :--- | :--- |
| `node://test/` | `summary`/`full` | Test results and failure reports. |
| `node://profile/` | `summary` | Performance hot-spots and flamegraph summaries. |
| `node://trace/` | `stored`/`full` | Raw execution traces, filtered via `<get>`. |
| `node://coverage/` | `summary`/`full` | Code coverage maps and heatmaps. |

## 2. Tools

### 2.1 `<node action="..." path="..."/>` (Action Tool)

Triggers a new Node.js execution.

- **Attributes**:
  - `action`: Required. One of `test`, `profile`, `trace`, `coverage`.
  - `path`: Required. The target file or directory.
  - `args`: Optional. Additional CLI arguments for the tool.
- **Behavior**: Records an entry at `node://{action}/{path}` with **status 202**.
- **Client Role**: The client observes the 202 entry, executes the corresponding
  command, and calls `resolve` with the result.

### 2.2 `<get path="node://..."/>` (Investigation Tool)

Leverages the core `get` tool to retrieve and filter Node artifacts.

- **Behavior**:
  - If a **body filter** is provided, the plugin uses `hedberg` to filter the
    raw artifact (even if `stored`) and returns matching lines.
  - If no body is provided, returns the `full` projection of the artifact.

## 3. Data Model & Projections

### 3.1 Projections (`full`/`summary`)

- **Summary**: Concise stats (e.g., "Tests: 12 passed, 2 failed").
- **Full**:
  - **Test**: TAP or JSON output formatted for LLM readability.
  - **Trace**: A navigable window into the execution log, showing causal links.
  - **Profile**: A list of the top 10 functions by CPU/memory usage.

### 3.2 Metadata (Attributes)

Entries resolved by the client should include:
- `exitCode`: Integer.
- `duration`: Execution time in ms.
- `tracePath`: (For tests) A link to a corresponding `node://trace/` entry.

## 4. Integration Points

- **Scheme**: `node` (category: `logging`).
- **Hooks**:
  - `handler`: Processes the `<node>` tool call.
  - `full`/`summary`: Projects the `node://` artifacts.
  - `on("handler")` for `get`: Centralized filtering logic for namespaced paths.
- **Utilities**: Uses `core.hooks.hedberg` for pattern matching and filtering.
